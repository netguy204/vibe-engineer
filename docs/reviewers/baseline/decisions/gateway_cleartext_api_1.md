## Criteria Assessment

### Criterion 1: GET endpoint returns decrypted messages for a valid token

- **Status**: satisfied
- **Evidence**: `workers/leader-board/src/swarm-do.ts#handleGatewayAPI` GET branch (lines 248-323) reads messages via `storage.readAfterBatch()`, decrypts each body with `decryptMessage()`, and returns JSON `{messages: [{position, body, sent_at}]}`. Tested in `test/gateway-api.test.ts` — "GET retrieves decrypted messages" and "full round-trip" tests verify correct plaintext recovery.

### Criterion 2: POST endpoint encrypts and stores messages for a valid token

- **Status**: satisfied
- **Evidence**: `workers/leader-board/src/swarm-do.ts#handleGatewayAPI` POST branch (lines 326-368) parses `{body}` from JSON, encrypts via `encryptMessage()`, stores via `storage.appendMessage()`, wakes watchers/polls, and returns `{position, channel}`. Body size validation at 1MB. Tested in "POST stores an encrypted message" and the WebSocket interop tests prove encryption compatibility.

### Criterion 3: Invalid/revoked tokens return 401

- **Status**: satisfied
- **Evidence**: `handleGatewayAPI` (lines 228-245) hashes the token, looks up the gateway key record, and returns 401 `{error: "Invalid or revoked token"}` if not found. Crypto errors (corrupted blob, wrong token) are caught and also return 401. Tested by "invalid token returns 401" and "revoked token returns 401" (which DELETEs the key blob first).

### Criterion 4: Token is used transiently — never stored in plaintext

- **Status**: satisfied
- **Evidence**: The token arrives in the URL path, is used to compute `hashToken(token)` for lookup, then `decryptBlob(blob, token)` to recover the seed, then `deriveSymmetricKey(seed)` to get the symmetric key. All derived key material is local variables scoped to `handleGatewayAPI` and discarded after the response. No caching, no persistence of plaintext key material. The `PendingPoll` struct does hold `symmetricKey` for long-poll duration, but this is in-memory only and cleaned up on resolve/timeout.

### Criterion 5: Long-poll variant blocks until messages arrive or timeout

- **Status**: satisfied
- **Evidence**: `handleGatewayAPI` GET branch checks for `?wait` parameter (clamped to 1-60s), creates a Promise with a `setTimeout` deadline, stores a `PendingPoll` in the per-channel map. `wakePendingPolls()` (lines 379-415) resolves pending polls when POST writes new messages. Timeout cleanup via `removePendingPoll()`. Three long-poll tests verify: immediate return when messages exist, blocking + wake on new message, and empty return on timeout (~1s).

## Additional Observations

- **Cross-language crypto compatibility**: Test vectors generated by Python (`generate-test-vectors.py`) and embedded in `test/gateway-crypto.test.ts` verify that the TypeScript crypto module (SHA-256 hashing, NaCl secretbox, Ed25519→Curve25519 clamping, HKDF-SHA256) produces byte-identical results to the Python `src/board/crypto.py`.
- **WebSocket↔gateway interop**: Two tests verify that messages written via the gateway can be read via WebSocket (and vice versa), confirming encryption compatibility between the two paths.
- **Channel name validation**: Uses the same regex (`/^[a-zA-Z0-9_-]{1,128}$/`) as the WebSocket protocol, returning 400 for invalid names.
- **Backreferences**: Properly placed in all new/modified files per project conventions.
- **All 83 tests pass** across all 10 test files, including all pre-existing tests.
