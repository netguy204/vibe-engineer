# Chunk Goal

## Minor Goal

Move the synchronous `oracle.analyze_conflict()` call off the async event loop in the scheduler's dispatch tick.

In `src/orchestrator/scheduler.py`, the `_dispatch_tick` method holds `self._lock` (an `asyncio.Lock`) for its entire duration. Within that locked section, `_check_conflicts` calls `oracle.analyze_conflict()` synchronously (line 1115), which performs file I/O (reading GOAL.md, PLAN.md, and frontmatter for each chunk pair via `Chunks` and `StateStore`). This blocks the entire event loop for the duration of the analysis, preventing other coroutines from making progress and adding latency to the dispatch cycle.

The fix is to run the oracle analysis via `asyncio.to_thread()` so file I/O executes in a thread pool. Because `asyncio.Lock` is not thread-safe and must not be held across an `await` that yields to a thread, the conflict analysis must be restructured: either release the lock before the threaded call and re-acquire after, or pre-compute all needed conflict analyses outside the locked dispatch tick and use only the cached results inside it.

This advances the trunk goal of maintaining workflow health over time by ensuring the orchestrator's dispatch loop remains responsive under load, particularly when many chunks require pairwise conflict analysis.

## Success Criteria

- `oracle.analyze_conflict()` is invoked via `asyncio.to_thread()` (or equivalent) so that its file I/O runs in a thread pool rather than blocking the event loop
- `self._lock` is not held while the thread pool executes the oracle analysis; the lock is either released before the threaded call and re-acquired after, or the analysis is restructured to run as a pre-computation step outside the locked `_dispatch_tick` section
- Cached conflict verdicts (`work_unit.conflict_verdicts`) continue to be checked and populated correctly, preserving the existing short-circuit behavior where cached results skip the oracle entirely
- The `_dispatch_tick` method retains its existing correctness guarantees: no work unit is dispatched while blocked, and conflict verdicts are still persisted on both sides of the pair
- All existing tests in `tests/` pass without modification (or with minimal adaptation to the new async structure)
- No new race conditions are introduced: shared state accessed by the oracle thread (e.g., `StateStore`, `Chunks`) is either thread-safe or copied before the thread call
